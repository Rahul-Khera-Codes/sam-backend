## Issue 1 — Voice Agent Cannot Retrieve Available Slots

**Severity:** Critical — core booking flow broken
**Status:** ✅ Fixed — `feature/available-slots-tools`
**Reported by:** Sam (live test during meeting)
**Heard during call:** *"It seems we're having an issue retrieving available time slots"*

### Root Cause (confirmed in code)

**File:** `agent/agent.py:333–337`

```python
date_err = _validate_booking_datetime(
    self._supabase, self._business_id, self._location_id, date, "00:00"  # ← BUG
)
```

`get_available_slots` passed `"00:00"` (midnight) to `_validate_booking_datetime`. That function checks if the time falls within business hours. Business hours are typically 9am–5pm, so midnight failed the check.

### Fix Applied
- Added `_validate_booking_date(supabase, business_id, location_id, date)` — date-only validation, no time check
- `get_available_slots` now uses `_validate_booking_date` instead
- Added `find_next_available_slot` tool — proactively offers next available without asking caller to guess a date
- 21 tests, all passing

## Issue 6 — Quick Agent Control Toggle Does NOT Actually Disable the Agent

**Severity:** 🔴 Critical — the ON/OFF toggle is completely non-functional
**Status:** ❌ Not fixed — needs implementation
**Discovered:** 2026-05-15 during code review

### What Users Expect
Flipping the toggle OFF in CS Scheduler → Quick Agent Control should stop the AI from answering calls.

### What Actually Happens

When the toggle is flipped, the frontend calls `PUT /settings/agent/state` which writes `is_active = false` to the `agent_state` table. **The voice agent never reads this table.** At call time, the agent only checks `custom_schedules.is_agent_disabled`. The toggle has zero effect on real agent behaviour.

**Confirmed in `agent/agent.py`:**
```python
# Only check that exists — custom schedule
_cs = _fetch_active_custom_schedule(supabase, business_id, location_id)
if _cs and _cs.get("is_agent_disabled"):
    disabled_by_schedule = _cs

# agent_state.is_active is NEVER read
```

### Spec for Fix

**Files to modify:**
- `agent/supabase_helpers.py` — add `_fetch_agent_state` helper
- `agent/agent.py` — check `is_active` at session start

---

#### New helper: `_fetch_agent_state`

Add to `agent/supabase_helpers.py`:

```python
def _fetch_agent_state(
    supabase,
    business_id: str,
    location_id: str | None,
) -> dict | None:
    """
    Returns the agent_state row for this business/location.
    Returns None if no row exists — caller should treat missing row as is_active=True.
    """
    try:
        q = (
            supabase.table("agent_state")
            .select("is_active, toggled_at, toggled_by")
            .eq("business_id", business_id)
        )
        if location_id:
            q = q.eq("location_id", location_id)
        else:
            q = q.is_("location_id", "null")
        r = q.limit(1).execute()
        return r.data[0] if r.data else None
    except Exception as e:
        logger.warning("Failed to fetch agent_state: %s", e)
        return None
```
